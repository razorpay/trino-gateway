# Query Entity

An audit/tracking record created whenever the trino-gateway router successfully proxies a SQL query to a Trino backend. Each Query captures what was sent, who sent it, which backend served it, and which group was used for routing -- forming the gateway's query history and enabling backend lookup for in-flight queries.

---

## Decisions

### D1: Fire-and-Forget Async Query Saving

**Context:** After routing a query to a Trino backend, the gateway needs to record the routing decision. Saving synchronously would add latency to every query response.
**Decision:** Query records are saved in a background goroutine (`go func()`) after the response is proxied back to the client.
**Alternatives considered:**
- **Synchronous save:** Rejected because it adds DB write latency to the critical query response path.
- **Batch/queue writes:** Rejected as over-engineering for the write volume; single goroutine per query is sufficient.
**Trade-offs:**
- Gained: Zero added latency on the query response path.
- Lost: Query save failures are silently logged -- no retry, no guarantee of persistence. The query history is best-effort, not authoritative.
**Code:** `response.go:ProcessResponse()` -- the `go func()` block at line context of `QueryRequest` case.
**Revisit if:** Query history becomes a billing or compliance requirement where every record must be persisted.

### D2: Upsert via Find-then-Create/Update (CreateOrUpdateQuery)

**Context:** The query ID comes from the Trino server response (extracted from the JSON body). The same query could be re-submitted or the router may process the same query ID again.
**Decision:** Check if a record exists with `Find()`, then `Update()` or `Create()` accordingly.
**Alternatives considered:**
- **MySQL UPSERT (INSERT ... ON DUPLICATE KEY UPDATE):** Would be more efficient but the spine ORM abstraction doesn't expose upsert natively.
- **Always INSERT, ignore duplicates:** Rejected because updates to existing query records (e.g., from subsequent requests in a multi-statement flow) would be lost.
**Trade-offs:**
- Gained: Works within the spine ORM abstraction; handles both new and repeated query IDs.
- Lost: Two DB round-trips (SELECT + INSERT/UPDATE) per save. Acceptable because saves are async.
**Code:** `core.go:CreateOrUpdateQuery()`
**Revisit if:** Query volume grows significantly and the double round-trip becomes a DB bottleneck.

### D3: FindBackendForQuery as a Dedicated Endpoint

**Context:** Clients (or the UI) need to determine which Trino backend is handling a specific query -- useful for debugging slow/stuck queries or for direct backend access.
**Decision:** Expose `FindBackendForQuery` as a separate Twirp RPC that returns just the `backend_id` and `group_id` for a given query ID.
**Alternatives considered:**
- **Use GetQuery and let the caller extract backend_id:** Would work but leaks unnecessary data and requires the caller to know the field name.
**Trade-offs:**
- Gained: Purpose-built API for the most common lookup pattern; clean contract.
- Lost: Minor code duplication (both call `GetQuery` internally).
**Code:** `server.go:FindBackendForQuery()`
**Revisit if:** More query metadata lookups are needed -- may consolidate into a richer GetQuery response.

## Non-Obvious Constraints

### Query Text Size Limit

**Rule:** `text` column is VARCHAR(500), increased from the original VARCHAR(255).
**Why:** Real-world Trino SQL statements frequently exceeded 255 characters. The migration (`20220107205304_increase_query_text_size`) doubled the limit. Still truncates queries longer than 500 chars at the DB level (silently, depending on MySQL strict mode).
**Enforced at:** MySQL column constraint.
**Examples:** Complex analytics queries with multiple JOINs and subqueries commonly exceed 255 chars but most fit within 500.

### Query ID Comes from the Trino Server, Not the Gateway

**Rule:** The query `id` is extracted from the Trino backend's HTTP response body (`response.go:extractQueryIdFromServerResponse()`), not generated by the gateway.
**Why:** This is Trino's native query ID (e.g., `20210805_195304_00001_abcde`), which is what users and monitoring tools reference. Using a gateway-generated ID would create a mapping problem.
**Enforced at:** `response.go:extractQueryIdFromServerResponse()` parses the JSON response for the `Id` field.

### Validation is Effectively Disabled

**Rule:** `ValidateMultiFetchRequest()` always returns nil. All struct-level validation in `validation.go` is commented out.
**Why:** Likely deferred during initial development and never re-enabled. The fetcher package handles basic pagination bounds.
**Enforced at:** `validation.go:ValidateMultiFetchRequest()` -- the validation framework imports (ozzo-validation) are commented out.

### Foreign Keys to Group and Backend

**Rule:** `queries.group_id` references `groups_(id)` and `queries.backend_id` references `backends(id)`. Both are nullable.
**Why:** A query record is meaningless without knowing which group and backend served it, but nullable FKs allow partial records (e.g., if group resolution succeeded but backend selection failed -- though this path currently doesn't save a record).
**Enforced at:** Bootstrap migration `20210805195304_bootstrap.go:Up20210805195304()`.

### Frontend Pagination is Client-Side

**Rule:** The query list UI (`querylistview.go:NewQueryListView()`) fetches up to 1000 items in one call and paginates client-side with a default of 100 items per page.
**Why:** Simpler implementation; the commented-out server-side pagination code suggests this was intentional but may be revisited.
**Enforced at:** `querylistview.go:populateItems()` -- `MaxItems: 1000`, `ItemsPerPage: 100`.

## Flow Map

### Create/Save Query (from Router)

| Flow | Trigger | Key Functions | Decision Points | Outcome |
|------|---------|---------------|-----------------|---------|
| Happy path (most traffic) | Trino backend returns 2xx for a QueryRequest | `response.go:ProcessResponse()` -> `response.go:extractQueryIdFromServerResponse()` -> `server.go:CreateOrUpdateQuery()` -> `core.go:CreateOrUpdateQuery()` | DP1, DP2 | Query record persisted |
| Save failure (rare) | DB error during create/update | `core.go:CreateOrUpdateQuery()` -> `repo/query.go:Create()` or `Update()` | -- | Error logged, query lost (fire-and-forget) |
| Non-query request (most traffic) | ApiRequest or UiRequest type | `response.go:ProcessResponse()` | DP1 | No query record created |

**Decision Points:**
- **DP1: Request type switch** -- Only `QueryRequest` types trigger query saving. API and UI requests are passed through without recording. Why: Only SQL query submissions carry a Trino query ID worth tracking.
- **DP2: Exists check (upsert)** -- `core.go:CreateOrUpdateQuery()` checks if the query ID already exists. Why: The same Trino query ID can appear in multiple response cycles (e.g., status polling), and we want the latest metadata.

### FindBackendForQuery

| Flow | Trigger | Key Functions | Decision Points | Outcome |
|------|---------|---------------|-----------------|---------|
| Happy path (common) | Twirp RPC call with query_id | `server.go:FindBackendForQuery()` -> `core.go:GetQuery()` -> `repo/query.go:Find()` | -- | Returns backend_id and group_id |
| Query not found (rare) | Invalid or expired query_id | `repo/query.go:Find()` | -- | Error returned |

### ListQueries

| Flow | Trigger | Key Functions | Decision Points | Outcome |
|------|---------|---------------|-----------------|---------|
| Happy path (common) | Twirp RPC or frontend call with filters | `server.go:ListQueries()` -> `validation.go:ValidateMultiFetchRequest()` -> `core.go:FindMany()` | DP3 | Filtered, paginated query list |

**Decision Points:**
- **DP3: Filter construction** -- `core.go:FindMany()` builds conditions from optional username, backend_id, group_id filters using struct tags with `omitempty`. Why: Allows flexible querying without requiring all filter fields -- empty strings are excluded from the WHERE clause via the `structs` library's `omitempty` json tag behavior.

## Service Contracts

### Who Creates Query Records

| Producer | Mechanism | Contract |
|----------|-----------|----------|
| Router (`response.go:ProcessResponse()`) | Async goroutine calling `gatewayApiClient.Query.CreateOrUpdateQuery()` via Twirp | Sends query text, client IP, group_id, backend_id, username, server_host. Query ID extracted from Trino response JSON. Fire-and-forget -- no retry on failure. |

### Who Reads Query Records

| Consumer | Mechanism | Contract |
|----------|-----------|----------|
| Frontend UI (querylistview) | `GetQueries()` via gateway API client | Expects paginated list with username filter. Fetches up to 1000 records. |
| FindBackendForQuery callers | Twirp RPC | Given a Trino query ID, returns which backend_id and group_id handled it. Used for debugging routing. |

### Dependencies

| Dependency | What Breaks If Down |
|------------|---------------------|
| MySQL (queries table) | All query CRUD fails. Async saves silently fail (logged). FindBackendForQuery returns errors. |
| Group entity (groups_ table) | FK constraint means queries referencing deleted groups would fail on insert. |
| Backend entity (backends table) | FK constraint means queries referencing deleted backends would fail on insert. |
